## 📋 一、需求概述

### 1.1 用户原始想法

用户在使用 Cline 让大模型生成 Logo 和促销海报时，发现大模型可以直接用 **HTML/CSS/SVG** 等代码方式生成精美的动态设计作品（如 `opera-logo.html` 动态 Logo 和 `618-poster` 促销海报）。

由此产生构想：在 PromptCraft 中增加一个**"动态设计"功能模块**，作为设计助手的**下游延伸**，让用户能够：
- 在设计助手中完成设计参数配置和提示词生成后
- 一键跳转到动态设计页面，自动携带设计参数
- 通过 LLM 将设计描述转化为 HTML/CSS/SVG 动态代码
- 实时预览、编辑、迭代修改动态效果
- 导出为 HTML / SVG / PNG / GIF 等格式

### 1.2 核心设计原则：双入口架构

> **⚠️ v3 关键更新：双入口设计**  
> 动态设计 TAB 是一个**可独立使用的功能页面**，同时也兼任**设计助手的下游渲染引擎**。

#### 🚪 双入口模式

| 入口 | 使用场景 | 数据来源 | UI 状态 |
|------|----------|----------|---------|
| **独立入口** | 用户直接点击"🎬 动态设计" Tab | 用户手动输入 | 显示简化输入面板 |
| **下游入口** | 设计助手生成结果后点击"🎬 生成动态设计" | 自动继承设计参数 | 隐藏输入面板，直接进入预览 |

#### 📊 两种入口的 UI 差异

```
┌─────────────────────────────────────────────────────────────────┐
│                     独立入口（完整模式）                           │
├─────────────────────────────────────────────────────────────────┤
│  ┌─ 简化输入面板 ──────────────────────────────────────────┐    │
│  │ 品牌名称: [________]  主色: [#____]  动画类型: [下拉▼]  │    │
│  │                           [🎬 生成动态设计]             │    │
│  └──────────────────────────────────────────────────────────┘    │
│  ┌─ 预览区 ─────────┐ ┌─ 代码编辑区 ─────────────────────┐    │
│  │                  │ │ <style>                         │    │
│  │   [动态预览]     │ │   @keyframes fadeIn {...}       │    │
│  │                  │ │ </style>                        │    │
│  └──────────────────┘ └──────────────────────────────────┘    │
│  ┌─ 导出工具栏 ─────────────────────────────────────────────┐    │
│  │ [📋 复制代码] [📥 HTML] [🖼️ PNG] [🎞️ GIF] [🔄 重新生成] │    │
│  └──────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    下游入口（精简模式）                           │
├─────────────────────────────────────────────────────────────────┤
│  ┌─ 继承参数展示 ─────────────────────────────────────────┐    │
│  │ 📋 来自设计助手 | Logo: TechCorp | 风格: 科技感 | ...  │    │
│  └──────────────────────────────────────────────────────────┘    │
│  （其余 UI 与独立入口相同）                                       │
└─────────────────────────────────────────────────────────────────┘
```

#### 🎯 设计原则

| 关注点 | 描述 |
|--------|------|
| 🚪 双入口支持 | 独立使用 + 设计助手下游，两种模式无缝切换 |
| 🎯 功能专注 | 核心能力：预览 + 代码编辑 + 迭代修改 + 多格式导出 |
| 🔗 数据流向 | 下游模式：设计助手 → 动态设计（单向传递设计参数） |
| 💰 Token 节约 | 适配超轻量模型（如 0.6B Qwen），控制生成 Token |
| ⚡ 渐进实现 | 不应过度复杂，分阶段开发 |

| ⭐⭐⭐⭐ | **活动海报** (`event-poster`) | 倒计时/日期动态效果 | Phase 14 扩展 |
| ⭐⭐⭐⭐ | **广告创意** (`ad-creative`) | Banner 广告常需动画 | Phase 14 扩展 |
| ⭐⭐⭐ | **品牌海报** (`brand-poster`) | 品牌元素可加呼吸灯效果 | Phase 15 可选 |

| ⭐⭐ | **书籍封面** (`book-cover`) | 翻页动效可选，非核心 | Phase 15 可选 |

| ❌ | **包装设计** (`packaging-design`) | 3D 实物，非 Web 内容 | 不适合 |
| ❌ | **商插/童插/概念艺术** | 复杂插画，超出 CSS 能力 | 不适合 |
| ❌ | **电商主图** (`ecommerce-product`) | 产品摄影为主 | 不适合 |
| ❌ | **UI 界面** (`ui-interface-design`) | 需交互原型工具 | 不适合 |

## 🔍 二、行业调研与参考项目

### 2.1 核心参考项目

#### 🍎 Apple Keyframer（最高参考价值）

| 项目 | 详情 |
|------|------|
| **名称** | [Keyframer](https://github.com/apple/ml-keyframer) |
| **作者** | Apple Machine Learning Research |
| **论文** | "Keyframer: Empowering Animation Design using Large Language Models" (ICLR) |
| **Star** | 1K+ |
| **核心思路** | 用户上传 SVG 图像 → 输入自然语言描述 → GPT-4 生成 CSS 动画代码 → 实时预览 |

**关键设计理念（对本项目极具参考价值）：**
- ✅ **SVG + CSS 动画**：LLM 生成 CSS `@keyframes` 动画代码，而非完整应用
- ✅ **迭代式设计**：支持"探索"（生成多个变体比较）和"精炼"（追加提示词修改）
- ✅ **双重编辑**：可通过自然语言提示词修改，也可直接编辑 CSS 代码
- ✅ **轻量级**：只生成 CSS 代码（通常几百 token），而非完整的 React/Next.js 应用
- ⚠️ **局限性**：仅动画已有 SVG，不能从零生成 SVG 图形本身

**核心架构启示：**
```
用户输入 SVG + 自然语言描述
       ↓
   LLM 生成 CSS 动画代码
       ↓
   CSS 注入到 SVG 中渲染预览
       ↓
   用户可编辑代码或追加提示词
```

通过 Context7 文档深入研究了以下技术库：

| 库名称 | 大小 | 特点 | CDN 可用 | 适合本项目 |
|--------|------|------|---------|-----------|

| **SVG SMIL 动画** | 0KB | 原生 SVG 动画，声明式 | N/A | ⭐⭐⭐⭐ |
| **Anime.js** | ~17KB | 强大的 SVG/CSS 动画，时间线控制 | ✅ | ⭐⭐⭐⭐ |

| **GSAP** | ~25KB | 专业级动画，TextPlugin/SVG 支持 | ✅ | ⭐⭐⭐ |
| **Lottie-web** | ~250KB | After Effects 动画播放 | ✅ | ⭐⭐ |
| **html2canvas** | ~40KB | DOM 截图导出为图片 | ✅ | ⭐⭐⭐⭐（导出用） |

**推荐策略：以纯 CSS/SVG 动画为主，Anime.js 为辅**

理由：
1. 纯 CSS 动画是 LLM 最擅长生成的，Token 消耗最少
2. Anime.js 轻量且功能强大，适合更复杂的 Logo 动画（如描边动画、路径动画）
3. html2canvas 用于导出功能
4. 不建议引入 GSAP（商业许可限制）和 Lottie（需要 After Effects 工作流）

### 3.1.1 数据传递流程

设计助手 → 动态设计的**完整数据链**：

```javascript
// 设计助手中点击 "🎬 生成动态设计" 时传递的数据
const dynamicDesignInput = {
  // 来自 selectedDesignPreset
  designType: selectedDesignPreset.id,       // 如 'logo-design'
  designTypeName: selectedDesignPreset.name,  // 如 'Logo 设计'
  
  // 来自 designVariables (用户填写的表单值)
  variables: { ...designVariables },          // 如 { brand_name: 'TechFlow', style: '科技感', colors: '#6366F1,#06B6D4' }
  
  // 来自 designResult (AI 已生成的提示词)
  structuredPrompt: designResult.structuredPrompt,  // 中文结构化描述
  finalPrompt: designResult.finalPrompt,            // 英文最终提示词
  keywords: designResult.keywords,                  // 关键词
  matchedSkills: designResult.matchedSkills,        // 匹配的专业技能
  
  // 额外
  additionalInput: designAdditionalInput            // 用户补充说明
};
```

**这意味着：动态设计页面不需要任何输入表单，所有设计信息从设计助手自动传入。**

### 3.2 前端架构详解

#### 3.2.1 iframe 沙箱预览（核心组件）

**这是最关键的技术决策。** 参考 Claude Artifacts 的做法，使用 `sandbox` 属性的 iframe 来安全渲染 LLM 生成的代码。

```html
<!-- 安全沙箱 iframe -->
<iframe 
  id="design-preview"
  sandbox="allow-scripts"
  class="w-full h-96 border rounded-lg"
  srcdoc="<!-- LLM 生成的 HTML/CSS/SVG 代码 -->"
></iframe>
```

**安全性考虑：**

- `sandbox` 属性限制 iframe 权限
- 不允许 `allow-forms`（防止表单提交）
- 不允许 `allow-top-navigation`（防止页面跳转）
- 不允许 `allow-popups`（防止弹窗）
- 使用 `srcdoc` 而非 `src`（避免网络请求）

**为什么选择 iframe 而不是直接渲染：**

1. **安全隔离** - LLM 生成的代码可能有意外行为，iframe 提供天然沙箱
2. **样式隔离** - 生成的 CSS 不会影响主应用的样式
3. **完整渲染** - iframe 支持完整的 HTML 文档，包括 `<style>`、`<script>`
4. **导出便利** - iframe 内的内容就是最终的独立 HTML 文件

```javascript
// gif.js 浏览器端 GIF 录制
async function exportGIF(iframeElement, duration = 3000) {
  const GIF = window.GIF; // CDN 加载
  const gif = new GIF({
    workers: 2,
    quality: 10,
    width: 400,
    height: 300
  });
  
  const fps = 10;
  const totalFrames = (duration / 1000) * fps;
  
  for (let i = 0; i < totalFrames; i++) {
    // 每帧截取 iframe 内容
    const canvas = await html2canvas(iframeElement.contentDocument.body);
    gif.addFrame(canvas, { delay: 1000 / fps });
    await new Promise(r => setTimeout(r, 1000 / fps));
  }
  
  gif.on('finished', (blob) => {
    const a = document.createElement('a');
    a.href = URL.createObjectURL(blob);
    a.download = 'design.gif';
    a.click();
  });
  gif.render();
}
```

### 3.4 Token 优化策略（重点！）

这是用户最关心的问题之一。以下是多层次的 Token 优化方案：

#### 策略一：精简的系统提示词（~200-400 tokens）

```
你是 HTML/CSS 动画代码生成专家。
生成要求：
1. 输出完整独立的 HTML 文件（包含内联 CSS）
2. 仅使用纯 CSS 动画（@keyframes + animation）
3. 使用 SVG 绘制图形元素
4. 代码简洁，不超过 100 行
5. 不使用外部依赖
6. 背景默认透明或白色
输出格式：直接输出 HTML 代码，不要解释。
```

对比传统图像提示词系统提示词（1000-3000 tokens），**节省 70-85%**。

#### 策略二：模板 + 变量填充（~300-800 tokens）

预置 HTML/CSS 模板骨架，LLM 只需生成差异化的部分：

```html
<!-- 预置的 Logo 动画模板骨架 -->
<!DOCTYPE html>
<html>
<head><style>
  body { display:flex; justify-content:center; align-items:center; 
         min-height:100vh; margin:0; background:{{background}}; }
  .logo { /* {{logo_styles}} */ }
  @keyframes logoAnim { /* {{animation_keyframes}} */ }
  .logo { animation: logoAnim {{duration}} {{easing}} {{iteration}}; }
</style></head>
<body>
  {{svg_content}}
</body>
</html>
```

LLM 只需填充 `{{}}` 变量，Token 消耗**降低 60%**。

#### 策略三：分步生成

将复杂设计拆分为多个轻量请求：
1. **第一步**：生成静态 SVG 结构（300-500 tokens）
2. **第二步**：为 SVG 添加 CSS 动画（200-400 tokens）
3. **第三步**：微调颜色/字体/细节（100-200 tokens）

每步控制在超轻量模型的 Token 限制内。

#### 策略四：复用已有压缩服务

整合 Phase 12 已实现的 `PromptCompressorService`，为代码生成场景添加专门的压缩等级。

#### Token 消耗对比表

| 场景 | 传统图像提示词 | 动态设计代码 | 节省比 |
|------|--------------|-------------|--------|
| 简单 Logo | 系统提示词 2000 + 输出 500 | 系统 300 + 输出 800 | ~56% |
| 宣传海报 | 系统提示词 2500 + 输出 600 | 系统 400 + 输出 1500 | ~39% |
| 动态图标 | 系统提示词 2000 + 输出 400 | 系统 200 + 输出 400 | ~75% |

> **结论**：简单的动态设计实际上并不比生成图像提示词消耗更多 Token！关键是限制生成代码的复杂度。

### 6.2 用户工作流（修订版）

```
唯一工作流：从设计助手进入
─────────────────────────
① 在"� 设计助手"Tab 正常使用：
   选择设计类型 → 填写品牌名、风格、配色 → 生成提示词

② 生成结果出来后，在按钮行看到新增的"🎬 生成动态设计"按钮

③ 点击按钮 → 系统自动将设计参数打包 → 切换到"🎬 动态设计"Tab

④ 动态设计页面自动显示设计摘要（只读），用户只需：
   - 选择动画效果（渐显/弹跳/描边等）
   - 选择生成模式（完整生成 or 模板填充）
   - 点击"🎬 生成动态设计"

⑤ iframe 中实时预览动态效果

⑥ 如需修改：
   a) 在迭代输入框描述修改 → 应用修改（AI 迭代）
   b) 直接编辑代码 → 即时同步预览

⑦ 满意后导出：HTML / SVG / PNG / GIF

⑧ 如需修改设计参数（品牌名/风格/配色）→ 点"🔙 返回修改参数" → 回到设计助手
```

**关键区别（相比 v1 方案）：**

| 对比项 | v1 方案（已废弃） | v2 方案（当前） |

| 品牌名/风格/配色输入 | 动态设计页面有完整的输入面板 | ❌ 无，来自设计助手 |
| 提示词输入 | 动态设计页面有独立的提示词框 | ❌ 无，复用设计助手的提示词 |

### 7.1 完整工作流示例：动态 Logo

**第一步：在设计助手中操作（已有功能）**

- 选择预设：Logo 设计
- 填写参数：品牌名 "PromptCraft"，风格 "科技感"，配色 "#6366F1, #06B6D4"
- 点击"生成专业提示词" → 生成结果出现
- 点击结果面板中的"🎬 生成动态设计"按钮

**第二步：自动跳转到动态设计 Tab**

系统传递的数据：

```javascript
dynamicDesignInput = {
  designType: 'logo-design',
  designTypeName: 'Logo 设计',
  variables: { brand_name: 'PromptCraft', style: '科技感', colors: '#6366F1,#06B6D4' },
  structuredPrompt: '科技感现代Logo...',
  finalPrompt: 'A tech-modern logo for "PromptCraft"...',
  keywords: ['科技', '现代', '渐变'],
  matchedSkills: ['色彩理论大师']
}
```

**第三步：用户在动态设计页面操作**

- 查看设计摘要（只读）：✅ Logo 设计 · PromptCraft · 科技感 · ■#6366F1 ■#06B6D4

- 选择生成模式：完整生成
- 点击"🎬 生成动态设计"

## 💡 八、创新点与差异化

### 8.1 与现有工具的差异

| 特性 | v0.dev / Bolt.new | Claude Artifacts | **PromptCraft 动态设计** |
|------|-------------------|-----------------|------------------------|
| 目标 | 通用 Web 开发 | 通用独立应用 | **专注设计领域** |
| Token 消耗 | 很高（数万） | 高（数千） | **极低（数百）** |
| 本地模型 | ❌ | ❌ | **✅ 支持 Ollama** |
| 设计知识 | ❌ | ❌ | **✅ Skills 系统** |

| 超轻量模型适配 | ❌ | ❌ | **✅ 模板+参数模式** |

### 8.2 PromptCraft 的独特优势

1. **集成式设计流** - 设计助手 → 动态设计一键衔接，零重复操作
2. **Token 极致优化** - 模板模式下仅需 <100 tokens 完成一个设计
3. **双轨产出** - 同一个设计需求，既可生成 AI 图像提示词，也可生成动态代码
4. **本地友好** - 唯一支持超轻量本地模型的动态设计工具
5. **知识增强** - Skills 系统提供设计知识，比通用 AI 工具更专业
6. **多格式导出** - HTML / SVG / PNG / GIF 四种导出格式全覆盖

## 📝 九、结论与建议

### 9.1 总体评价

**完全可行，且与现有项目高度契合！**

核心理由：

1. **技术成熟** - SVG + CSS 动画 + iframe 沙箱都是成熟技术，有大量参考项目
2. **LLM 擅长** - HTML/CSS 代码是 LLM 最擅长生成的内容之一
3. **架构复用** - 80% 以上的后端基础设施可直接复用
4. **Token 可控** - 通过模板化策略，Token 消耗甚至低于图像提示词
5. **差异化明显** - 没有其他工具专注于"设计领域 + 本地模型 + 极简动态设计"
6. **零功能重复** - v2 方案确保动态设计页面不重复设计助手已有功能

### 9.2 建议的实施路径

```text
Phase 13 MVP（17.5h）:
├── 设计助手结果面板 → "🎬 生成动态设计"入口按钮
├── 动态设计 Tab（设计摘要 + 动画配置 + iframe 预览）
├── 代码编辑（textarea）+ 迭代修改
├── HTML 导出
└── 单元测试

Phase 14 增强（16h）:
├── SVG / PNG / GIF 导出（按需加载库）
├── 预置模板库（5-10 个高质量模板）
├── 模板+参数混合模式（适配超轻量模型）
└── 历史记录集成

Phase 15 高级（13h）:
├── CodeMirror 代码编辑器
├── 预览尺寸切换
├── CSS 动画专家 Skill
└── 动画参数可视化调节
```

### 9.3 不建议做的

| 不建议 | 原因 |
|--------|------|
| ❌ 视频生成 | Token 消耗巨大，效果难以保证 |
| ❌ 复杂交互应用 | 超出极简设计的定位 |
| ❌ React/Next.js 代码生成 | 过于复杂，不适合本项目技术栈 |
| ❌ 引入 WebContainers | 过度工程化，iframe 已足够 |
| ❌ 动态设计页面添加独立输入面板 | 与设计助手功能重复 |

### 9.4 最终推荐

**推荐立即启动 Phase 13 MVP 开发**，预计 17.5 小时即可完成核心功能，风险可控，收益明显。

关键改进（v2 相比 v1）：动态设计作为设计助手的**下游渲染引擎**，而非独立的设计工具，确保功能不重复、数据流清晰、用户体验连贯。
