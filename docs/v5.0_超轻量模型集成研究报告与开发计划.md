## 1. 执行摘要

本报告为 PromptCraft v5.0 版本提供完整的技术研究和开发规划。核心目标是**在应用内直接嵌入超轻量 AI 模型**，实现离线可用的智能提示词辅助和内容安全审查，同时保持应用轻量、流畅。

### 核心决策总览

| 决策项 | 推荐方案 | 体积 | 理由 |
|--------|----------|------|------|
| 提示词增强模型 | **Qwen3-0.6B** (GGUF Q4_K_M) | ~462 MB | 最新一代，思考/非思考双模式，中英文极佳 |
| 安全审查模型 | **Tiny-toxic-detector** (ONNX) | ~10 MB | 2.1M参数，90%+准确率，极低开销 |
| 文本生成运行时 | **node-llama-cpp** | ~15 MB (npm) | Node.js 最优 GGUF 推理引擎 |
| 分类任务运行时 | **Transformers.js** | ~5 MB (npm) | 纯 JS/WASM，零原生依赖，跨平台 |
| **两模型总体积** | **~477 MB** | ✅ **远低于 800MB 限制** | |

### 2.2 当前 AI 能力

| 功能 | 实现方式 | 依赖 |
|------|----------|------|
| AI 提示词生成 | `llmService.call()` → 外部 OpenAI 兼容 API | 需要外部 LLM |
| AI 提示词改进 | `improverService.improveWithAI()` | 需要外部 LLM |
| 规则提示词改进 | `improverService.improve()` | 纯规则，无 AI |
| 规则提示词分析 | `improverService.analyze()` | 纯规则，无 AI |
| 提示词压缩 | `promptCompressorService` | 纯规则，无 AI |
| 安全审查 | **❌ 不存在** | - |

### 2.3 核心痛点

1. **完全依赖外部 API**：无 API Key 则所有 AI 功能不可用
2. **无安全审查机制**：用户输入/输出无任何内容安全检查
3. **低配电脑体验差**：无离线 AI 能力，规则引擎效果有限
4. **项目名称冲突**："PromptCraft" 在 GitHub 上有大量同名项目

## 4. 推荐模型方案

### 4.1 方案一：双模型架构（⭐ 推荐）

采用 **"生成模型 + 分类模型"** 双引擎架构，各司其职：

```
┌─────────────────────────────────────────────────┐
│             模型层 (Model Layer)                 │
│                                                 │
│  ┌─────────────────┐  ┌──────────────────────┐  │
│  │   Qwen3-0.6B    │  │  Tiny-toxic-detector │  │
│  │   (GGUF Q4_K_M) │  │      (ONNX)          │  │
│  │    ~462 MB       │  │      ~10 MB           │  │
│  │                  │  │                       │  │
│  │  via             │  │  via                  │  │
│  │  node-llama-cpp  │  │  Transformers.js      │  │
│  │                  │  │                       │  │
│  │  用途:           │  │  用途:                │  │
│  │  • 离线提示词增强│  │  • 输入安全检测        │  │
│  │  • 离线提示词改进│  │  • 输出安全检测        │  │
│  │  • 智能意图分类  │  │  • 实时内容过滤        │  │
│  │  • 智能翻译辅助  │  │                       │  │
│  └─────────────────┘  └──────────────────────┘  │
│                                                 │
│  总体积: ~477 MB  |  内存占用: ~600-800 MB       │
└─────────────────────────────────────────────────┘
```

**为什么选择 Qwen3-0.6B：**

1. **思考/非思考双模式**：复杂任务启用思考模式（Chain-of-Thought），简单任务用非思考模式实现低延迟响应。这对我们的项目非常关键——提示词改进需要推理，但快速分类不需要。
2. **中文能力顶尖**：支持 100+ 语言，中文理解和生成能力在同级别模型中最强。我们的项目面向中文用户，中文输入→英文提示词是核心功能。
3. **体积合理**：GGUF Q4_K_M 量化后仅 462MB，远低于 800MB 限制。
4. **上下文窗口大**：32K token 上下文，足够处理复杂的系统提示词 + Skills 上下文 + 用户输入。
5. **Apache-2.0 许可**：完全自由商用，无法律风险。
6. **社区活跃度最高**：Qwen 系列在 HuggingFace 下载量位居前列，社区支持最好。

**为什么选择 Tiny-toxic-detector：**

1. **极致轻量**：仅 2.1M 参数，约 10MB，对应用体积几乎零影响。
2. **精度足够**：ToxiGen 数据集 90.97% 准确率，Jigsaw 数据集 86.98%，对于前置过滤已经足够。
3. **推理速度极快**：CPU 上毫秒级响应，不会影响用户体验。
4. **Transformers.js 兼容**：纯 JS/WASM 运行，无需原生编译，打包友好。

### 4.2 方案二：单模型架构（备选）

如果追求极致简洁，可以仅使用 Qwen3-0.6B 兼顾生成和安全审查：

```
Qwen3-0.6B 同时负责:
├── 提示词增强 (非思考模式，低延迟)
├── 提示词改进 (思考模式，高质量)
├── 内容安全审查 (非思考模式 + 系统提示词约束)
└── 意图分类 (非思考模式)
```

**优点**：只需维护一个模型。  
**缺点**：安全审查不够专业，且每次审查都要占用生成模型的推理资源，生成和审查不能并行。

### 4.3 最终推荐

**强烈推荐方案一（双模型架构）**。原因：

1. 分类模型（10MB）的额外体积可以忽略不计
2. 安全审查和内容生成可以**并行**执行，不互相阻塞
3. 专用分类模型的安全检测**准确率更高**
4. Transformers.js 运行分类模型**零原生依赖**，打包更简单
5. 关注点分离，代码更清晰，维护性更好

### 5.2 具体集成位置分析

#### 位置 A：提示词改进（`improverService.js`）⭐ 最高优先级

**现状**：
- `improve()` 方法是纯规则引擎，效果有限
- `improveWithAI()` 依赖外部 LLM API

**集成方案**：
```
新增: improveWithLocal(prompt) 
├── 使用 Qwen3-0.6B 思考模式
├── 调用 promptCompressorService 压缩上下文
├── 结合 analyze() 的评分数据生成改进版本
└── 无外部 API 时自动降级到此方法
```

**集成文件**：`backend/services/improverService.js`  
**改动范围**：新增 `improveWithLocal()` 方法，修改 `improveWithAI()` 添加降级逻辑

#### 位置 B：安全审查中间件（全新）⭐ 最高优先级

**现状**：完全没有安全审查

**集成方案**：
```
[新] backend/middleware/safetyMiddleware.js
├── 全局 Express 中间件
├── 拦截所有 POST 请求中的用户文本输入
├── 使用 Tiny-toxic-detector 进行毒性分类
├── 分数 > 阈值则拒绝请求
└── 记录安全事件日志

[新] backend/services/safetyService.js  
├── SafetyService 类
├── checkInput(text) → { safe: boolean, scores: {...}, details: string }
├── checkOutput(text) → { safe: boolean, ... }
├── getConfig() → 安全策略配置
└── 支持自定义敏感词库
```

**请求处理流程（集成后）**：
```
用户输入 → [安全审查中间件] → 路由处理 → [输出安全检查] → 返回结果
              │                                    │
              ├─ 安全 → 继续                       ├─ 安全 → 返回
              └─ 不安全 → 返回 403 + 提示          └─ 不安全 → 过滤/重新生成
```

#### 位置 C：离线提示词生成（`promptGenerator.js`）⭐ 高优先级

**现状**：
- `generate()` 是模板匹配，不需要 AI
- `generateAI()` 完全依赖外部 API

**集成方案**：
```
新增: generateLocal(input, scenarioId)
├── 使用 Qwen3-0.6B 非思考模式（低延迟）
├── 复用 expertPrompts 中的系统提示词
├── 通过 promptCompressorService 压缩到 ULTRA 等级
└── 生成质量介于模板和外部 API 之间
```

#### 位置 D：智能意图分类（全新）⭐ 中优先级

**现状**：设计角色选择完全靠用户手动选

**集成方案**：
```
[新] backend/services/intentService.js
├── classifyDesignIntent(userInput)
├── 使用 Qwen3-0.6B 非思考模式
├── 输入用户描述，输出最匹配的设计角色 ID
├── 前端显示 "智能推荐" 标签
└── 用户可选择采纳或忽略
```

**示例**：
```
输入: "帮我设计一个咖啡品牌的标志"
输出: { recommended: "logo-design", confidence: 0.95, alternatives: ["brand-poster"] }
```

#### 位置 E：中英翻译辅助（增强现有功能）⭐ 中优先级

**现状**：中文→英文提示词翻译完全依赖外部 API

**集成方案**：
```
在 promptGenerator.generateAI() 或 generateLocal() 中:
├── 用户中文输入 → Qwen3-0.6B 翻译为英文关键词
├── 利用模型的多语言能力进行术语翻译
└── 不替代外部 API，作为离线降级方案
```

#### 位置 F：输出安全后检查⭐ 中优先级

**集成位置**：`backend/routes/prompts.js` 中所有生成相关路由的响应阶段

```
路由响应前:
├── AI 生成完成 → 调用 safetyService.checkOutput(result)
├── 安全 → 正常返回
├── 不安全 → 标记 warning / 过滤关键词 / 重新生成
└── 在响应中添加 safety 字段供前端展示
```

### 5.3 分层降级策略

```
┌─────────────────────────────────────────────┐
│  用户触发 AI 功能                            │
│                                             │
│  Layer 1: 外部 API (最高质量)               │
│  ├── 有 API Key + 网络连接 → 使用外部 API    │
│  │                                           │
│  Layer 2: 本地模型 (良好质量)               │
│  ├── 无 API / 离线 → 使用 Qwen3-0.6B        │
│  │                                           │
│  Layer 3: 规则引擎 (基础质量)               │
│  └── 模型未加载 → 使用规则引擎 (improve等)   │
└─────────────────────────────────────────────┘
```

这意味着：**用户无论什么配置，都能使用 AI 功能**——只是质量有所不同。

## 6. 安全审查系统设计

### 6.1 多层安全防线

```
┌────────────────────────────────────────────────────────────┐
│                    安全审查系统架构                         │
│                                                            │
│  Layer 1: 关键词预过滤（正则表达式，<1ms）                 │
│  ├── 硬编码敏感词列表                                      │
│  ├── 正则匹配 → 直接拒绝                                   │
│  └── 开销: 零                                              │
│                                                            │

└────────────────────────────────────────────────────────────┘
```

### 6.2 设计领域的特殊考量

我们的项目是**设计提示词生成器**，这带来一些独特的安全挑战：

1. **误判风险高**：设计相关的正常词汇可能被误判为不安全（如"裸色"、"nude tone"是合法的色彩术语）
2. **图像生成上下文**：我们生成的提示词最终会用于图像生成，NSFW 图像提示词需要特别关注
3. **多语言混合**：用户中文输入 + 英文输出，需要双语安全检测

**解决方案**：
```javascript
// 设计术语白名单示例
const DESIGN_SAFE_TERMS = [
  'nude tone', 'nude color', '裸色',
  'flesh color', '肤色',
  'body language', '肢体语言',
  'figure drawing', '人体素描',
  'anatomy', '解剖学',
  // ... 更多设计领域安全术语
];
```

## 7. 模型微调与优化策略

### 7.1 是否需要微调？

| 场景 | 基础模型表现 | 是否需要微调 | 建议 |
|------|-------------|-------------|------|
| 通用提示词改进 | ⭐⭐⭐⭐ 良好 | ❌ 不需要 | Qwen3-0.6B 原生能力已够用 |
| 设计领域提示词生成 | ⭐⭐⭐ 中等 | ⚠️ 可选 | 用 LoRA 微调可提升设计专业性 |
| 中文→英文设计术语翻译 | ⭐⭐⭐⭐ 良好 | ❌ 不需要 | 原生多语言能力强 |
| 设计意图分类 | ⭐⭐⭐ 中等 | ✅ 建议 | 微调可显著提升分类准确率 |
| 安全内容检测 | ⭐⭐⭐⭐ 良好 | ⚠️ 后续优化 | Tiny-toxic-detector 已够用 |

### 7.2 推荐的微调方案

#### 阶段一（v5.0）：System Prompt Engineering（无需微调）

**策略**：通过精心设计的系统提示词来引导 Qwen3-0.6B 的行为，这是最低成本的"定制"方式。

```javascript
// 设计提示词增强的系统提示词
const DESIGN_ENHANCE_SYSTEM_PROMPT = `你是一个专业的设计提示词工程师。
你精通以下设计领域：Logo设计、海报设计、UI设计、插画设计等。
你的任务是将用户的中文设计需求转化为专业的英文AI图像生成提示词。

核心能力：
1. 精准理解设计意图
2. 使用专业设计术语（构图、光影、色调、材质等）
3. 针对 Midjourney/DALL-E/Stable Diffusion 优化输出格式
4. 保持简洁，控制在 200 tokens 以内

输出格式：直接输出英文提示词，无需解释。`;
```

**这种方式足以覆盖 v5.0 的需求**。

#### 阶段二（v5.1+）：LoRA 微调（可选升级）

如果 v5.0 上线后发现系统提示词方式效果不够好，则进行 LoRA 微调：

**微调数据来源**（已有的项目资产）：
```
1. data/templates.json        - 现有模板库（结构化提示词样本）
2. data/expert_prompts.json   - 专家提示词（高质量系统提示词）
3. data/professional-terms.json - 133+ 专业设计术语
4. skills/*/SKILL.md           - 4个专业知识模块
5. data/history.json           - 用户历史生成记录
6. data/favorites.json         - 用户收藏的优质提示词
```

**微调参数建议**：
```yaml
微调方法: QLoRA (4-bit quantized LoRA)
基础模型: Qwen3-0.6B
LoRA rank: 16
LoRA alpha: 32
学习率: 2e-4
Batch size: 4
训练步数: 500-1000
训练数据量: ~500-1000 条设计提示词对
框架: LLaMA-Factory (推荐) 或 Unsloth
硬件需求: 8GB+ VRAM GPU 或 16GB+ RAM CPU
训练时间: ~30分钟 (GPU) / ~2-4小时 (CPU)
输出格式: GGUF Q4_K_M (覆盖原始模型文件)
```

**微调后的模型体积不会增加**（LoRA 合并后重新量化为 GGUF）。

### 7.3 优化策略（运行时）

#### 内存优化
```javascript
// 延迟加载策略
class LocalModelManager {
  constructor() {
    this.generationModel = null;  // 按需加载
    this.safetyModel = null;      // 按需加载
  }

  async ensureGenerationModel() {
    if (!this.generationModel) {
      // 首次使用时才加载，约需 3-5 秒
      this.generationModel = await loadQwen3();
    }
    return this.generationModel;
  }

  // 内存不足时释放模型
  async unloadModel(type) {
    if (type === 'generation') {
      this.generationModel?.dispose();
      this.generationModel = null;
    }
  }
}
```

#### 推理优化
```
1. KV Cache：启用 key-value 缓存减少重复计算
2. 上下文预热：服务启动时预加载常用系统提示词
3. Batch 推理：多个安全检查请求合并执行
4. 流式输出：生成任务使用 streaming 模式
5. 压缩上下文：复用 promptCompressorService 的 ULTRA 模式
```

## 8. 项目重命名方案

### 8.1 命名重要性分析

当前 "PromptCraft" 在 GitHub 搜索中面临严重的名称冲突：
- 至少 **20+ 个同名项目**
- 涵盖各种语言和领域
- 甚至有同名项目也是做提示词管理的（如 Reddit 上的 PromptCraft/Prompt-Forge）

这导致：
- **SEO 极差**：搜索 "PromptCraft" 很难找到我们的项目
- **品牌混淆**：用户可能误进其他项目
- **Star 增长受限**：曝光度低

### 8.2 命名原则

1. **唯一性**：GitHub 搜索结果首位
2. **含义清晰**：名字能传达"设计 + AI + 提示词"
3. **易记忆**：2-3 个音节，朗朗上口
4. **国际化**：英文名，中英文用户都能理解
5. **域名可用**：`.dev` / `.io` 域名可注册（未来考虑）
6. **无商标风险**：不与知名品牌冲突

// backend/services/improverService.js (新增方法)

/**
 * 使用本地模型改进提示词（离线模式）
 * @param {string} prompt - 原始提示词
 * @returns {Promise<Object>} 改进结果
 */
async improveWithLocal(prompt) {
  const localModelManager = require('./localModelManager').getInstance();
  
  if (!localModelManager.isAvailable('qwen3-0.6b')) {
    // 降级到规则引擎
    return this.improve(prompt);
  }

  const analysis = this.analyze(prompt);
  const systemPrompt = this._getImproverSystemPrompt();
  const userPrompt = this._getImproverUserPrompt(prompt, analysis);
  
  // 压缩上下文以适应小模型
  const compressor = require('./promptCompressorService').promptCompressor;
  const compressed = compressor.compressSystemPrompt(systemPrompt, 2); // ULTRA mode
  
  try {
    const result = await localModelManager.generate(
      userPrompt,
      compressed,
      { 
        maxTokens: 512,
        temperature: 0.7,
        mode: 'thinking' // 启用思考模式提升改进质量
      }
    );
    
    const improved = this._parseAIResponse(result);
    const improvedAnalysis = this.analyze(improved.text);
    
    return {
      original: prompt,
      improved: improved.text,
      content: improved.text,
      analysis: { before: analysis, after: improvedAnalysis },
      aiExplanation: improved.explanation,
      source: 'local-model', // 标记来源
      improvement: {
        beforeScore: analysis.totalScore,
        afterScore: improvedAnalysis.totalScore,
        delta: improvedAnalysis.totalScore - analysis.totalScore
      }
    };
  } catch (error) {
    console.warn('Local model failed, falling back to rules:', error.message);
    return this.improve(prompt);
  }
}

### safetyMiddleware.js 示例

```javascript
// backend/middleware/safetyMiddleware.js

const safetyService = require('../services/safetyService').getInstance();

/**
 * 安全审查中间件
 * 拦截所有包含用户文本的 POST 请求
 */
function safetyMiddleware(options = {}) {
  const {
    paths = ['/api/prompts'],  // 需要审查的路径前缀
    skipPaths = ['/api/safety', '/api/models'],  // 跳过的路径
    threshold = 0.7  // 毒性阈值
  } = options;

  return async (req, res, next) => {
    // 仅检查 POST/PUT 请求
    if (!['POST', 'PUT'].includes(req.method)) return next();
    
    // 路径过滤
    const shouldCheck = paths.some(p => req.path.startsWith(p));
    const shouldSkip = skipPaths.some(p => req.path.startsWith(p));
    if (!shouldCheck || shouldSkip) return next();
    
    // 提取文本内容
    const textFields = extractTextFields(req.body);
    if (textFields.length === 0) return next();
    
    try {
      const results = await Promise.all(
        textFields.map(text => safetyService.checkInput(text))
      );
      
      const unsafe = results.find(r => !r.safe);
      if (unsafe) {
        return res.status(403).json({
          success: false,
          error: {
            code: 'CONTENT_UNSAFE',
            message: '检测到不安全内容，请修改后重试',
            details: unsafe.details
          }
        });
      }
      
      next();
    } catch (error) {
      // 安全检查失败不应阻塞正常流程
      console.warn('Safety check error:', error.message);
      next();
    }
  };
}

function extractTextFields(body) {
  const texts = [];
  const fields = ['prompt', 'input', 'text', 'content', 'description', 'brandName'];
  for (const field of fields) {
    if (body[field] && typeof body[field] === 'string') {
      texts.push(body[field]);
    }
  }
  return texts;
}

module.exports = safetyMiddleware;
```
