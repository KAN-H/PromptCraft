## 一、项目背景

PromptAtelier 是一款专业设计提示词生成工作台，经过 Phase 1-12 的迭代，已具备完整的提示词生成、分类体系、Agent Skills、历史收藏、风格参考（Few-shot Learning）等功能。

**v5.0 的核心目标**：集成超轻量本地 AI 模型，使 PromptAtelier 具备**离线推理能力**，同时加入内容安全审查系统，保障生成内容的合规性。

### 1.1 为什么需要本地模型？

| 痛点 | 说明 |
|------|------|
| 网络依赖 | 当前所有 AI 功能都依赖外部 API，离线无法使用 |
| 成本问题 | 云端 API 调用产生持续费用 |
| 隐私顾虑 | 用户的创意内容需发送到第三方服务器 |
| 延迟不稳定 | 受网络状况影响，响应速度不可控 |
| 429 限流 | 部分 API 存在并发限制 |

### 1.2 技术选型

经过研究比较，选定以下超轻量方案：

| 组件 | 选型 | 说明 |
|------|------|------|
| 文本生成引擎 | `node-llama-cpp` v3.17+ | GGUF 格式推理，纯 Node.js，无需 Python |
| 分类引擎 | `@huggingface/transformers` v3.8+ | ONNX 格式推理，浏览器/Node.js 双端 |
| 生成模型 | Qwen3-0.6B (Q4_K_M) | ~462MB，支持 100+ 语言，思考/非思考双模式 |
| 分类模型 | Tiny-Toxic-Detector | ~10MB，毒性内容分类 |
| 集成方式 | ESM dynamic `import()` | 两库均为 ESM-only，通过动态导入兼容 CommonJS |

### 1.3 架构设计原则

```
三层降级架构：
  优先级 1: 外部 API（云端推理，最高质量）
  优先级 2: 本地模型（离线可用，轻量快速）
  优先级 3: 规则引擎（纯算法兜底，零依赖）

设计理念：
  - 按需下载：模型不随项目分发，首次使用时下载
  - 延迟加载：模型按需加载到内存，空闲自动卸载
  - 优雅降级：任何层级不可用时自动退回下一层
  - 零阻塞：模型初始化/下载不阻塞主功能
```

## 五、Phase 14 — 安全审查系统（✅ 后端已完成）

> 完成日期：2026-02-27（后端）

### 5.1 目标

构建内容安全审查系统，防止生成不当内容，同时避免设计领域术语的误报。

### 5.2 核心架构 — 三层审查

```
输入文本
  │
  ▼
Layer 1: 关键词预过滤（正则匹配）────→ 极低延迟，零依赖
  │ PASS
  ▼
Layer 2: AI 毒性分类（Tiny-Toxic-Detector）──→ 本地 ONNX 模型
  │ PASS
  ▼
Layer 3: 语义深度审查（Qwen3-0.6B）────→ 可选，高级场景
  │
  ▼
审查结果: PASS / WARN / BLOCK
```

**关键创新 — 设计安全词白名单**：

设计领域存在大量容易被安全系统误报的术语，例如：
- `nude tone`（裸色调） → 不是不当内容
- `bleed`（出血位） → 印刷术语
- `body copy`（正文内容） → 排版术语
- `kill fee`（中止费用） → 行业术语
- `strip`（条状设计） → 设计元素

系统内置 **87 个设计安全术语** 白名单，在关键词过滤前先检查上下文，大幅降低误报率。

### 5.3 核心实现

#### 5.3.1 SafetyService（`backend/services/safetyService.js`，616 行）

**核心方法**：

| 方法 | 功能 |
|------|------|
| `initialize()` | 加载配置，编译正则表达式 |
| `check(text, options)` | 执行完整安全审查（三层链） |
| `_checkKeywords(text)` | Layer 1: 关键词正则匹配 |
| `_checkAIClassify(text)` | Layer 2: AI 毒性分类 |
| `_checkSemantic(text, context)` | Layer 3: 语义深度审查 |
| `_isDesignSafeText(text)` | 设计安全词白名单检查 |
| `_determineCategory(text)` | 自动判定内容类别 |
| `getConfig()` / `updateConfig(updates)` | 配置管理 |

**分类与处理策略**：

| 分类 | 阈值 | 动作 | 说明 |
|------|------|------|------|
| toxic | 0.7 | block | 有毒内容 → 直接拦截 |
| insult | 0.7 | warn | 侮辱内容 → 警告 |
| threat | 0.6 | block | 威胁内容 → 直接拦截 |
| hate | 0.6 | block | 仇恨内容 → 直接拦截 |
| sexual | 0.8 | warn | 性暗示 → 警告（设计领域阈值较高） |

#### 5.3.2 安全中间件（`backend/middleware/safetyMiddleware.js`，164 行）

**工作流程**：
```
请求进入
  │
  ├─ 跳过路径检查 → /api/safety/*, /health → 放行
  ├─ 方法检查 → GET/HEAD/OPTIONS → 放行
  ├─ 保护路径检查 → /api/prompts/*, /api/skills/* → 需审查
  │
  ▼
提取文本字段（prompt, text, content, description, name 等）
  │
  ▼
调用 safetyService.check()
  │
  ├─ BLOCK → 返回 403 + 错误详情
  ├─ WARN  → 设置 X-Safety-Warning header → 放行
  └─ PASS  → 放行
  
异常处理：中间件错误不阻塞请求（降级放行）
```

**导出测试辅助函数**：`_shouldSkipPath`, `_shouldCheckMethod`, `_isProtectedPath`, `_extractTextFields`

### 8.2 新增功能

#### 8.2.1 空闲自动卸载

`localModelManager.js` 新增模型空闲自动卸载机制：

| 方法 | 功能 |
|------|------|
| `setIdleTimeout(minutes)` | 设置空闲超时（默认 30 分钟） |
| `getIdleTimeout()` | 获取当前超时设置 |
| `startIdleCheck()` | 启动定时检查（每 5 分钟） |
| `stopIdleCheck()` | 停止定时检查 |
| `_performIdleCheck()` | 内部：检查并卸载超时模型 |

- `generate()` / `classify()` 自动更新 `_lastUsedAt` 时间戳
- 超时后发出 `model:idle-unload` 事件
- `shutdown()` 自动停止定时器，`.unref()` 避免阻止进程退出

#### 8.2.2 错误自动恢复

`generate()` 方法增加 context/sequence 损坏检测与自动重建：

```
生成异常 → 正则检测关键词(sequence|context|disposed)
  ├─ 匹配 → dispose 旧 context → 创建新 context → 分配新 sequence → 重试
  └─ 不匹配 → 正常抛出错误
```

## 十一、风险与缓解

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| 模型下载失败 | 本地推理不可用 | 支持手动导入、多镜像源 |
| 内存不足 | 模型加载失败 | 延迟加载、自动卸载、内存监控 |
| 推理速度慢 | 用户体验差 | 提示词压缩、量化模型、超时控制 |
| 安全误报 | 正常设计词被拦截 | 设计安全词白名单、阈值可调 |
| ESM 兼容性 | 动态导入可能失败 | 运行时检测、降级到规则引擎 |
